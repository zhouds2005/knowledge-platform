# 部门知识共享平台 · 总体设计

> 日期：2026-06-19
> 状态：已确认
> 关联：后续详见 `docs/superpowers/plans/` 目录下的实施计划

## 项目背景

大型国企下属单位，约 300 人，实际种子用户 10-50 人。Phase 1 目标：部署到内网、让种子用户实际使用、跑通完整的文档上线 → 共享 → 检索 → 通知闭环。如果效果好，后续扩展为全企业职工使用的统一知识共享平台。

**核心痛点：**
- 部门内部文档分散，找不到
- 跨部门共享权限管不住
- 文档不能在线预览
- 文档版本不受控
- 知识散落在个人手中，无法统一检索
- 企业共用文档不能第一时间让员工知晓

**Phase 1 范围：** 单位公用文档的存储、共享、发布、预览、版本控制、搜索、通知。

## 一、认证与用户模型

### 现状

系统使用自建 session-cookie 认证（`server/middleware/auth.ts`），用户密码存本地 `users.password_hash`。

### 方案

引入 **MaxKey 开源 IAM**，走 OIDC 协议做统一认证。本地不再管理密码。

**改造三层：**

1. **认证中间件**：`server/middleware/auth.ts` 换成 OIDC 校验。未登录用户重定向到 MaxKey 登录页；回调后解析 id_token，查/建本地用户记录，生成 session。
2. **用户表收窄**：`users` 表新增 `externalId` 字段存 MaxKey 用户 ID。密码字段保留但不使用（兼容期），后续 migration 移除。首次登录时从 MaxKey claims 同步 displayName、email、departmentId。
3. **前端改造**：`AuthProvider` 不再调本地 `/api/login`，检测 401 后跳转 MaxKey 授权页。登出清理本地 session + 调 MaxKey 登出端点。

**RBAC 保持不变：** 角色（admin/space_admin/reviewer/viewer）在本地 `users.role` 管理，不依赖 MaxKey。理由：MaxKey 角色粒度不够细。

## 二、用户权限

### 全局角色

| 角色 | 权限 |
|------|------|
| admin（管理员） | 全平台所有权限 |
| reviewer（审核员） | 审核分配给自己的文档 |
| viewer（普通用户） | 按可见范围阅读 |

### 空间管理员（新增）

新增 `space_members` 表，记录用户在空间内的角色（admin / member）。一个用户可在空间 A 是管理员、空间 B 是普通成员。

**字段：** userId, spaceId, role（admin | member）

**权限逻辑：** 在 `canEdit` / `canRead` 中增加判断——如果用户是该文档所属空间的管理员，自动有空间内所有文档的读写权。全局管理员依然能管所有内容。

### 文档级权限（保持现有）

- 每篇文档有可见范围：space / department / public
- 草稿和审核中的内容只有作者和审核员可见
- `objectPermissions` 表支持精细授权（指定某人可看/可编辑/可审核）

## 三、文件存储

### 现状

系统已有 Nextcloud 集成代码（`server/lib/nextcloud.ts`、`server/routers/nextcloud.ts`），且有 Nextcloud 实例可用。

### 方案

Nextcloud 作为唯一存储后端，平台只存元数据。

**存储路径（Nextcloud 上）：** `/知识共享平台/{space_name}/{knowledge_id}/{version}/{filename}`

**上传流程：** 浏览器 → Express（multer 临时） → Nextcloud API 上传 → 删临时文件。数据库记录 Nextcloud 路径、大小、hash、file_id。

**下载/预览：** 走后端代理，不直连 Nextcloud。后端校验权限后才转发文件流。好处是权限完全由平台控制。

**大文件优化（Phase 1 不做）：** 上传改为后台排队（用户传完即提示成功，服务器异步搬文件）；大文件下载给直链绕过中转。

## 四、文件预览

Phase 1 支持格式：PDF、图片（jpg/png/gif）、纯文本（txt/md）、Office 文档（Word/Excel/PPT）。

| 格式 | 实现方式 |
|------|---------|
| PDF、图片 | 浏览器原生渲染，服务器直接转发文件流 |
| Office 文档 | 服务器安装 LibreOffice，后台转 PDF 后传给浏览器 |
| 纯文本 | 直接显示在网页上 |

预览入口：文档列表中点文件，弹出预览窗口（不跳页），同时提供"下载"和"版本历史"按钮。Office 首次转换需几秒，同一版本缓存后秒开。

---

## 五、统一检索

基于 PostgreSQL 全文搜索（`search_vector` 字段已存在），搜文档标题 + 文档内容。

- 搜索结果按权限过滤（只显示用户有权看的内容）
- 搜索框在页面顶部，实时出结果，按相关度排序
- 结果显示：标题、所属空间、更新时间、匹配摘要

## 六、通知机制

**Phase 1：** 仅站内通知（页面右上角铃铛图标）。

通知类型：
- 空间有新文档发布
- 你的文档被审核通过/驳回
- 给你指派了审核任务

数据库：`notifications` 表已有（字段：userId, type, objectId, message, read, createdAt）。

**后续（Phase 2+）：** 接入微信通知，另开一轮设计。

## 七、版本控制

### 现有机制（保留）

- `knowledgeObjects` 表通过 `sourceTable` + `sourceId` 组合识别同一文档的不同版本
- 编辑已发布文档时，插入新行，版本号 +1，状态重置为草稿。旧版保持不变
- 草稿/审核中内容原地修改，不产生新版本
- 版本历史按 `sourceTable` + `sourceId` 查询，按版本号倒序

### 需要补齐

1. **文件附件版本追踪**：新增 `knowledge_file_versions` 表，记录每版对应的 Nextcloud 文件路径、大小、hash
2. **版本变更说明**：`knowledgeObjects` 表加 `changeNote` 字段，记录"此版本改了什么"
3. **回滚接口**：选历史版本 → 将该版本状态设为 published，当前版本归为 archived，新版本号 = 历史版本号 + 1

## 八、UI 重新设计

### 附加：Wiki 富文本编辑器

**问题：** 现有 Wiki 编辑器功能弱，仅简单文本输入框。

**方案：** 引入 Tiptap（基于 ProseMirror），替换 Wiki 编辑页的输入框为富文本编辑器。

Phase 1 支持格式：
- 标题（H1-H3）
- 加粗、斜体、下划线
- 有序/无序/待办列表
- 插入图片（从 Nextcloud 选取或上传）
- 插入表格
- 引用块、分隔线

Phase 1 不做：
- 多人同时编辑（需 WebSocket + CRDT，放 Phase 2）
- @ 提及（需和通知系统打通，放 Phase 2）

**实现：** 安装 `@tiptap/react` 及扩展包，编写 `src/components/wiki/RichEditor.tsx` 组件，替换现有 Wiki 编辑页。内容存 HTML 格式，复用 `knowledgeObjects.description` 字段。后端无需改动。


### 色彩体系（70-20-10）

- 主色（70%）：深石板灰 `#1e293b`（侧边栏/顶栏）+ 暖米白 `#faf8f5`（内容背景）
- 辅色（20%）：暖灰 `#f1ede8`（卡片底色）
- 强调色（10%）：琥珀金 `#d97706`（按钮、链接、选中态）

### 字体

- 标题：Noto Serif SC（思源宋体）
- 正文：Noto Sans SC（思源黑体）

### 布局变化

- 首页增加统计行（知识总量、今日更新、待审核、空间数）
- 知识列表从卡片堆叠改为行式列表
- 侧边栏深色背景，增加分组（"知识管理"/"系统管理"）
- 空间管理改为卡片网格

### 各页面设计

详见设计预览文件 `design-preview.html`，覆盖：工作台、文档列表、文档详情、Wiki、网盘、审核队列、用户管理、部门管理、空间管理、通知中心。

## 九、部署方案

**目标环境：** 内网 Ubuntu 20.04+ 服务器

**依赖组件：**
- Node.js 18+
- PostgreSQL 14+
- Nginx（反向代理 + 静态文件）
- LibreOffice（Office 预览转换）
- pm2（进程守护，自动重启）

**部署步骤：**
1. 代码拉取到 `/opt/knowledge-platform`
2. `npm install` 装依赖
3. 配置 `.env`（DATABASE_URL、NEXTCLOUD_*、MAXKEY_*、SESSION_SECRET）
4. `npm run db:push` 建表
5. `npm run build:all` 构建
6. `pm2 start dist/index.js --name knowledge-platform`
7. Nginx 配置反向代理

**已有脚本复用：** `scripts/deploy.sh`、`scripts/backup.sh`、`scripts/rollback.sh` 检查适配后直接使用。
