# 部门知识共享平台 — 完整操作手册 v1.0

---

## 一、系统架构概览

整个系统围绕"知识对象"运转。理解下面这层关系，所有操作就串起来了：

```
部门 (Department)                       ← 组织架构的顶层
  └─ 知识空间 (Knowledge Space)         ← 知识分类容器，隶属一个部门
       └─ 知识对象 (Knowledge Object)   ← 核心实体：文档/Wiki/网盘文件
            ├─ 版本链 (Versions)        ← 同 source 对象形成版本历史
            ├─ 关联图谱 (Graph)         ← 对象间的引用关系
            ├─ 权限授权 (Permissions)   ← 对象级的 view/edit/review 授予
            └─ 评审记录 (Review Record) ← 每一次通过/驳回的历史
```

**用户角色**（权限从高到低）：

| 角色 | 能做什么 |
|------|---------|
| admin | 管理用户/部门/空间、读写所有内容、审批所有对象 |
| editor | 创建和编辑自己的内容、在自己部门内读写已发布内容 |
| viewer | 只能读自己部门内的已发布内容 + 被显式授权的内容 |

## 三、核心业务流程

### 流程 1：知识对象的完整生命周期

这是系统最核心的流程，涉及创建、审核、发布，以及在此过程中的权限控制。

```
步骤                    操作者         发生的变化                 验证点
─────────────────────────────────────────────────────────────────────────────
1. 创建对象             editor         POST /api/knowledge        status="draft"
                                       ↓                         首页「我的草稿」出现
                                       插入 knowledge_objects     详情页显示「草稿」标签
                                       ↓
2. 提交审核             owner          点击「提交审核」            status 变为 pending_review
                                       ↓                         出现「审核中」黄色标签
                                       submitForReview()          owner 看到「撤回」按钮
                                       指定 reviewerId
                                       发送通知给 reviewer
                                       ↓
3. 审批通过             reviewer       点击「通过」                status 变为 published
                                       ↓                         出现「已发布」绿色标签
                                       approveReview()           旧 published 版本自动归档
                                       记录 review_records
                                       发送通知给 owner + 空间成员
                                       ↓
4. 审批驳回             reviewer       点击「驳回」+ 填写理由      status 变回 draft
                                       ↓                         回到第 2 步可重新提交
                                       rejectReview()
                                       发送通知给 owner
                                       ↓
5. 已发布对象编辑       editor         PUT /api/knowledge/:id     创建新 draft 版本
                                       ↓                         version+1
                                       旧 published 保持不动      版本历史中出现两条记录
                                       ↓
6. 删除（软归档）       editor/admin   DELETE /api/knowledge/:id  status 变为 archived
```

**测试这个流程的操作路径：**

> 登录 editor → 创建文档（选有默认审核人的空间）→ 详情页点「提交审核」→ 切换到 reviewer 账号 → 审核队列看到 → 点「通过」或「驳回」→ 切回 owner 账号查看状态变化 + 通知

### 流程 2：组织架构搭建（部门 → 空间 → 网盘）

部门、空间、网盘路径三者紧密关联：

```
创建部门                 创建空间                  上传网盘文件
POST /api/departments    POST /api/spaces          POST /api/drive/upload
│                        │                         │
├─ name: "技术部"        ├─ departmentId           ├─ 选中的空间有 nextcloudPath
├─ nextcloudPath 自动     ├─ name: "API文档"        ├─ 文件上传到 Nextcloud WebDAV
│  设为 "/技术部"         ├─ nextcloudPath 自动      ├─ 同时插入 knowledge_objects
│                         │   设为 "/技术部/API文档"  │   type="drive_file"
│                         │                         │   sourceTable="nextcloud_files"
│                         │                         │   departmentId/spaceId 继承
│                         │                         │
▼                         ▼                         ▼
departments 表            knowledge_spaces 表        knowledge_objects 表
nextcloud_path 写入       nextcloud_path 写入       + Nextcloud 文件系统
+ Nextcloud 创建文件夹    + Nextcloud 创建子文件夹
```

**关键逻辑：**
- 子部门的 nextcloudPath = 父路径 + "/" + 子部门名（通过递归查找祖先拼接）
- 空间的 nextcloudPath = 所在部门路径 + "/" + 空间名
- 上传文件创建的知识对象：type="drive_file"，自动归属到选中空间的部门和空间
- 网盘页面左侧树：先取部门列表，再取空间列表，按 departmentId 分组展示

**测试这个流程的操作路径：**

> admin 登录 → 部门管理创建「测试部」→ 验证目录中出现 `/测试部` → 空间管理创建「测试空间」选「测试部」→ 验证路径为 `/测试部/测试空间` → 网盘页左侧展开「测试部」→ 点击「测试空间」→ 右侧显示文件列表 → 上传文件 → 首页「最近更新」出现 drive_file 类型卡片

### 流程 3：权限控制体系

系统有三层权限，逐层叠加：

```
第 1 层：角色权限（代码写死）
  admin   → 所有对象可读可写可审
  editor  → 自己创建的对象可读写
  viewer  → 只能读

第 2 层：部门可见性（对象字段 visibility）
  space      → 只对同空间成员可见
  department → 只对同部门成员可见
  public     → 所有人可见

第 3 层：显式授权（object_permissions 表）
  对单个对象，可以授予特定用户 view / edit / review 权限
  这会叠加在第 1、2 层之上
```

**权限判断逻辑（server/lib/permissions.ts）：**

```
canRead:
  admin → 始终 true
  owner → 始终 true
  对象 status=draft 且非 owner → false（草稿只有 owner 和 admin 能看到）
  visibility=public → true
  同部门 → true
  有显式 view 授权 → true

canEdit:
  admin → true
  owner → true
  有显式 edit 授权 → true

canReview:
  admin → true
  对象 reviewerId == 当前用户 → true
  有显式 review 授权 → true
```

**测试权限的操作路径：**

> editor A 创建文档（visibility=department）→ editor B（同部门）能看到 ✓ → viewer C（不同部门）看不到 ✓ → A 在详情页点「权限」→ 添加 viewer C 的 view 授权 → C 刷新后能看到 ✓ → 撤销授权 → C 又看不到了 ✓

### 流程 4：收藏与浏览历史

这两个是独立的辅助功能，但都在首页有入口。

**收藏流程：**
```
知识详情页 → 点 Star 按钮 → POST /api/knowledge/:id/favorite
  ├─ 插入 user_favorites（唯一索引防重复）
  ├─ 按钮变「已收藏」，Star 实心黄色
  └─ 首页「我收藏的」刷新后出现

再次点击 → DELETE /api/knowledge/:id/favorite
  ├─ 删除 user_favorites 记录
  ├─ 按钮变回「收藏」，Star 空心
  └─ 首页「我收藏的」刷新后消失
```

**浏览历史流程：**
```
访问知识详情页 → recordView(userId, objectId) 自动触发
  ├─ 1 小时内重复访问 → 只更新时间戳，不新增记录
  ├─ 超过 1 小时 → 插入新记录
  └─ 超过 100 条 → 删除最旧的记录（trim）

首页「最近浏览」→ GET /api/history
清空 → DELETE /api/history
```

**收藏的跨用户隔离：**
- GET /api/favorites 只返回当前用户的收藏
- user_favorites 表的查询条件固定带上 `userId = req.user.id`
- 所以用户 A 收藏的东西，用户 B 完全看不到

**测试的操作路径：**

> 打开知识详情 → 点收藏 → 回首页看「我收藏的」出现了 → 再点已收藏取消 → 首页消失 → 再收藏 → 切换用户登录 → 该用户的收藏列表为空（隔离验证）→ 浏览几个知识详情页 → 首页「最近浏览」出现记录 → 点「清空」→ 记录消失

### 流程 5：关联图谱

知识对象之间可以建立引用关系：

```
详情页 → 点「关联」→ 搜索目标对象 → 选择 → POST /api/knowledge/:id/relation
  ├─ 插入 object_relations（sourceObjectId → targetObjectId）
  └─ 关联类型：references（引用）、attaches（附件）、belongs_to（归属）

详情页 → 点「图谱」→ GET /api/knowledge/:id/graph
  ├─ 查询所有与该对象相关的 relations
  ├─ 收集所有关联对象
  └─ 返回 nodes（节点）+ edges（边）供可视化渲染
```

**测试的操作路径：**

> 文档 A 详情页 → 点「关联」→ 搜索文档 B → 添加关联 → 点「图谱」→ 看到 A 和 B 的连线 → 回到文档 B 详情页 → 点「图谱」→ 也能看到与 A 的连线

---

### 流程 6：搜索

```
首页搜索框 → 输入关键词 → 选择类型过滤（可选）→ 点「搜索」
  ↓
GET /api/knowledge/search?q=xxx&type=document&space_id=xxx&status=xxx
  ↓
后端用 PostgreSQL ILIKE 在 title、description、tags 三个字段模糊匹配
  ↓
返回结果列表 → 首页切换到搜索结果视图 → 每张卡片可点击跳转
```

**测试的操作路径：**

> 首页搜索框输入关键词 → 回车 → 出现「搜索结果(N)」标题 + 卡片列表 → 选择类型过滤「文档」→ 结果只显示文档类型 → 搜索不存在的词 → 显示「没有找到相关内容」
